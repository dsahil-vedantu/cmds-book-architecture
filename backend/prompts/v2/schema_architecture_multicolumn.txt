## §3. PARENT ATTRIBUTION (the single rule that drives every placement)

### §3.0 Column reading order (MULTI-COLUMN — apply throughout)
On each page, read in this fixed order:
1. **2-column layout:** entire LEFT column top-to-bottom, THEN entire RIGHT column top-to-bottom.
2. **3-column layout:** entire LEFT column top-to-bottom, THEN entire MIDDLE column top-to-bottom, THEN entire RIGHT column top-to-bottom.
3. **Per-page layout can vary** — some pages may be single-column (covers, TOC, full-width tables/figures). Use the actual visual layout of EACH page.
4. **Full-width spanning headings (§4.7)** sit OUTSIDE the column flow — they apply to everything below them across all columns until the next spanning heading.
5. Facing pages: left page first (in its column order), then right page (in its column order).

### §3.1 The rule
Parent of any item = the MOST RECENT heading printed BEFORE the item in column reading order (§3.0). A heading whose title appears AFTER the item in column reading order — even on the same page — is NEVER the parent. Topic similarity is IRRELEVANT. Only physical position in column reading order.

### §3.2 Same-page binary (column-aware)
- Heading printed BEFORE item in column reading order → that heading is the parent. "Before" means: full-width spanning heading above the columns on this page, OR earlier in the same column above the item, OR in any earlier column on this page.
- Item printed BEFORE heading in column reading order → parent is the previous section (carry over from above / earlier page / earlier column).

### §3.2.5 Cat A nesting (HARD RULE)
Every Cat A subsection (EXAMPLE, EXERCISE, PRACTICE PROBLEM, etc.) MUST be a child of the immediately preceding Cat B (theory) section in document order — NEVER a sibling of theory sections at the same level.

### §3.2.6 Cat A parent — by HEADING POSITION, not text reference
A Cat A subsection (EXAMPLE, EXERCISE, PRACTICE PROBLEM) MUST nest under whichever theory section's page range CONTAINS the Cat A's OWN PRINTED HEADING — the physical position where "EXAMPLE 9.5" or "EXERCISE 1.6" is laid out as a section title in column reading order.

NEVER use a text reference position. If a heading says "see Example 9.5", that text reference is NOT where Example 9.5 lives — only the printed heading itself counts.

SAME-PAGE TIE-BREAK (column-aware):
When Cat A and the next theory heading share a page_start, use column reading order:
- Cat A printed BEFORE next heading in column reading order → belongs to PREVIOUS theory section
- Cat A printed AFTER current heading in column reading order → belongs to CURRENT section

Example: Page 3 left column has EXAMPLE 1.2 at top, then "9.3 Heading" mid-column. EXAMPLE 1.2 belongs to §9.2 (carry over from page 2 / earlier column), NOT to §9.3 (which starts AFTER Example 1.2 in column reading order).

### §3.3 Canonical example (multi-column)
2-column page: LEFT column shows EXAMPLE 3.2, EXAMPLE 3.3 at top. RIGHT column starts with heading "8.2 Types of Cells" then has more content. Page 3 ended with heading "Expressing the Fraction a/b as Percentage".

Column reading order: LEFT column first (EXAMPLE 3.2, 3.3) — these come BEFORE the right-column heading. Parent of EXAMPLE 3.2/3.3 = previous heading from page 3 ("Expressing the Fraction a/b as Percentage"), NOT "8.2 Types of Cells".

### §3.4 Cross-page / cross-column tie-break
When a child's title is printed near a page boundary (bottom of one page, top of next) or near a column transition (bottom of left col, top of right col), the title's `page_start` is the deciding line — the item belongs to whichever parent's heading appears LAST before that position in column reading order. Example: Example 3.8 title at bottom of left column on page 5; Point B's heading at top of left column on page 5 (above Example 3.8) → Point B precedes Example 3.8 in column reading order → Example 3.8 belongs to Point B.

## §4. WHAT IS A SECTION (include in schema)

### §4.0 HEADING ENUMERATION — SCAN EVERY PAGE TWICE

Before emitting the schema, do TWO passes over the PDF:

PASS 1 — IDENTIFY EVERY HEADING:
Mark every text on every page that is ANY of:
  - Larger font than surrounding body text
  - Bold/heavy weight while body text is regular
  - All-caps or small-caps while body text is mixed-case
  - Numbered prefix: "9.2", "1.5.3", "I.", "A.", "(a)"
  - Section keywords: "Introduction", "Summary", "Definition",
    "Examples", "Exercise", "Activity", "Key Ideas", "Remember",
    "Points to Remember", "Try It", "Practice Questions",
    "EXAMPLE N", "EXERCISE N.M"
  - Visual separators: lines above/below, indented block, boxed

PASS 2 — VERIFY COMPLETENESS:
For each page, list every heading found. If a page has substantial
content but ZERO headings detected → re-scan. Common patterns missed:
  - Unnumbered theory sub-headings ("Cathode Rays", "X-rays")
  - End-of-chapter banks ("JEE-NEET Special Wing", "Practice Set")
  - Numbered headings without clear visual distinction
  - Headings on the same page as figures or tables
  - Headings in a second/third column missed by reading only the
    leftmost column (apply §3.0 column reading order on every page)

SELF-CHECK before emitting:
  - Total schema sections + excluded_sections should cover EVERY page
  - If `total_pages = 46` but your sections only cover pages 1-36
    → you missed 10 pages of content. Re-scan pages 37-46.
  - Do NOT submit a schema with pages "missing" from coverage.

CRITICAL: heading completeness is REQUIRED. Missing a heading invalidates
your schema. The downstream verifier WILL catch missed headings via
font analysis — re-scan rather than ship incomplete.

### §4.1 Numbered headings
Chapter X / XA / XB (letter-prefixed); X.Y, X.Y.Z, X.Y.Z.W; Roman numeral chapters (I, II, III).

### §4.2 Unnumbered headings
Must satisfy BOTH: (1) visually distinct (bold OR larger font OR distinct color OR styled box), AND (2) introduces content. Common: Introduction, Summary, Recap, Quick Recap, Glossary, Index, Bibliography, References, Appendix A/B/C, Historical Background, Applications, About <Person>. Unnumbered chapter heading → level-1 section.

### §4.3 Lettered / numbered sub-headings (CRITICAL — preserves theory between examples)
A printed line starting with `(A)`, `(B)`, `(C)` OR `A.`, `B.` OR `1.`, `2.` becomes its OWN subsection of the surrounding theory heading when ALL hold:
- on its own line (not embedded in a paragraph),
- visually distinct (own line, possibly bold),
- followed by explanatory prose OR Cat A items (Examples/Exercises).

**Failure mode this prevents:** Pattern `Section 8.1 → A. <theory> → Example 8.1 → B. <theory> → Example 8.2`. If A. and B. are NOT subsections, the theory under each is LOST during extraction (merges into parent's intro). Always capture A. and B. as subsections.

Entry: `id=<parent>-a`, `title="(A) ..." verbatim`, `type="subsection"`, `content_types=["theory"]`, `level=parent.level+1`.

### §4.4 Wrapped titles
A heading spanning two printed lines is ONE heading — capture full title verbatim.

### §4.5 End-of-chapter teaching content (SECTIONS, not excluded)
Summary, Recap, Quick Recap, Key Concepts (heading for the whole block), Glossary, Historical context, Applications, About <Person> — all Cat B `["theory"]`. Bullet-point form still counts as teaching content.

### §4.6 Reference content
Index, Bibliography, References, Appendix A/B/C, Glossary → Cat B `["theory"]`. General rule: anything that is NOT a pure question bank / hints / solutions / answer keys is theory. When in doubt → theory.

### §4.7 SPANNING headings (multi-column)
A heading that visually spans the FULL page width (a banner crossing all columns) is the PARENT of everything below it across all columns on that page (and continuing pages until the next same-level heading).
Examples: a chapter heading "Chapter 8: Cell" banner at top of page above a 2-column body; a section heading "8.2 Types of Cells" full-width banner between two stretches of 2-column theory.
A heading constrained to ONE column (not full-width) applies only to content within that column flow until the next heading appears in reading order.

## §5. WHAT IS NEVER A SECTION (keep inline — no schema entry)

### §5.1 Recurring named callouts (Cat C, inline)
Did You Know, DO YOU KNOW, Note, Remember, Recall, Smart Tip, Konnect, Fun Fact, Fun Facts (even recurring across multiple pages), Key Concepts boxes, Key Points boxes, Thinking Corner, Errata, Note to Teacher, Progress Check (small recurring box), Let's Solve (small recurring box).
EXCEPTION — `Points to Remember` is ALWAYS its own Cat B section (not inline), even if short.

### §5.2 Subject-specific labels (inline)
Math: Theorem 5.1, Corollary 5.2, Proof, Lemma. Chemistry: Reaction 5.1, Equation 5.2. Biology: Process: Photosynthesis, Cycle: Krebs. Real teaching content but inline parts of the surrounding theory body — NOT sections.

### §5.3 Table / Figure labels (inline)
"Table 5.1: Properties of Metals" → inline. "Figure 4.3: Plant cell anatomy" → inline. Standalone full-page figure with no heading: record page in surrounding section's range; caption is inline body. NO separate schema entry.

### §5.4 Multi-step / multi-method internals (inline)
Activity Steps ("Step 1", "Step 2", "Step 3") → inline content of the Activity. Worked Example Methods ("Method 1", "Method 2") → inline content of the Example.

### §5.5 Page furniture (NEVER captured)
Page headers / running banners (chapter title repeated atop every page → ONE entry). Page numbers, footnotes, margin annotations, column footers (shared below all columns). Cross-references mid-paragraph ("see Section 5.2") → inline mentions.

Front-matter and back-matter housekeeping — copyright, ISBN, publisher info, author/editor bios, edition info, dedication, ceremonial foreword, acknowledgements, errata/corrigenda lists, blank pages → DO NOT emit. Note range briefly in `extraction_notes`. If foreword contains real teaching prose, treat as theory (§4.5).

### §5.6 MCQ options
`(A)`, `(B)`, `(C)`, `(D)` or `(1)`, `(2)`, `(3)`, `(4)` IMMEDIATELY following a question stem are answer options — NEVER sections. In multi-column layouts options may sit beside the stem (same column or adjacent) — still options.

### §5.7 TOC entries
Numbered lines under "Chapter Outline" / "Outline" / "Contents" / "Table of Contents" → TOC bullets, NEVER subsections. The TOC itself is ONE Cat B section with `expected_question_count=0` and `subsections=[]`.

### §5.8 Chapter intro styled callouts
Styled box immediately after chapter heading with intro prose → inline body of chapter, NOT a subsection.

### §5.9 Crosswords / Word Puzzles — STRICT, NEVER A SECTION
Crosswords, word puzzles, word searches, jumbles, sudoku, riddles, and similar puzzle blocks are ALWAYS Cat C inline body of the surrounding theory section. **NEVER emit a section entry for "Crossword" / "Word Puzzle" / etc.** They contain no extractable theory and no extractable questions — they're decorative learning aids. If you see a heading "Crossword" or "Word Puzzle" in the PDF: skip it, do NOT create a section in `sections[]` or `excluded_sections[]`. The surrounding theory section absorbs it as inline body (no special block needed; just continue extracting other content). Violating this rule produces a section with `expected_question_count=0` and pending status that will fail downstream.

### §5.10 Disambiguating rules
When a numbered/lettered line could be either subsection or inline:
- Followed by (A)(B)(C)(D) options OR ends with `?` → MCQ stem (inline).
- 4+ similarly-numbered items in a row under a Q-bank-style heading → individual questions (inline; parent gets `expected_question_count = N`). NEVER emit one subsection per question.
- Followed by real explanatory prose introducing a sub-concept → subsection.
- Single bullet under a TOC-style heading → TOC entry (inline, §5.7).
- Single styled box right after chapter heading → chapter intro callout (inline, §5.8).

Trust the printed question numbers over visual column boundaries.

## §8. EXCLUDED CONTENT (driven by §0.5 #4 POSITION rule)

`excluded_sections` is a FLAT array at top level, NOT nested under chapter.

### §8.1 What can be excluded (when no theory follows)
The decisive test: **is there ANY theory printed AFTER this item in column reading order?** If yes → INLINE Cat A (§8.3). If no → excluded.

Categories that can be excluded:
1. End-of-chapter question banks with named headings — see §7.1 Cat A list for all valid names.
2. Hints / Solutions / Answer Keys when STANDALONE and 1-line answer style (typically chapter end).
3. Truly anonymous numbered Q blocks (no heading, no teaching prose) — try §10 fallback first.
4. Structural exam-paper labels as parent banks — see §8.5.

### §8.2 What does NOT go excluded
Index, Bibliography, References, Glossary, Appendix A/B/C, Summary, Recap, Quick Recap, About <Person>, ANY teaching content. Default: **teaches → theory; pure Q-bank or 1-line answer key + no theory after → excluded**. When in doubt → theory.

### §8.3 Mixed-position Cat A — POSITION TRUMPS NAME (THE rule that fixes Exercise N.M misclassification)
A chapter can have BOTH inline Cat A AND end-of-chapter banks. Inline vs excluded is decided by position, NOT by the bank's name.

- **Inline Cat A** (any bank-named section followed by more theory in column reading order) → nested under its immediate preceding theory parent in `sections`. Applies to ALL bank names — Exercise N.M, Practice Problems, MCQs, Hints, Level 1, Worked Solutions, Numerical Problems, Assertion-Reason, etc.
- **Excluded** (true end-of-chapter, no theory after) → `excluded_sections`.

**Worked example (Real Numbers textbook):**
PDF column-reading order: §1.5 Theory → "Points to Remember" → "Exercise 1.6" → "ICT Corner" → §1.7 Theory → "Exercise 1.7" → "Unit Exercise" (chapter ends here, no theory after)

CORRECT placement:
```
sections: [chapter, subsections: [
  §1.5 [theory], subsections: [
    Points to Remember [theory],
    Exercise 1.6 [questions]   ← INLINE (theory follows: ICT Corner + §1.7)
  ],
  ICT Corner [theory],
  §1.7 [theory], subsections: [
    Exercise 1.7 [questions]   ← INLINE
  ]
]]
excluded_sections: [
  Unit Exercise [end of chapter, no theory after]
]
```

The rule: scan from the item forward in column reading order. If you hit ANY theory heading / theory section / theory sub-heading / ICT Corner / Points to Remember / Summary etc. before the chapter ends → INLINE. If only more question banks follow until chapter end → EXCLUDED.

### §8.4 Excluded entry shape (SUPPORTS NESTED SUBSECTIONS)
`title=verbatim`, `page_start`/`page_end`, `reason=short string`, `is_numbered=bool`, `expected_question_count=count`, `subsections=[]` (may contain nested ExcludedSection entries).

Each subsection inside an excluded entry uses the SAME ExcludedSection shape — NO `type`, NO `content_types`. Preserve printed sub-bank order. Parent's `expected_question_count` = sum of subsection counts. In multi-column layouts, count questions across all columns per §7.7.

Example — "Practice Questions" excluded bank with internal sub-categories:
```
{
  "title": "Practice Questions",
  "page_start": 18, "page_end": 25,
  "reason": "End-of-chapter practice block",
  "is_numbered": false,
  "expected_question_count": 60,
  "subsections": [
    {"title": "Very Short Answer Type", "page_start": 18, "page_end": 19,
     "is_numbered": false, "expected_question_count": 15, "subsections": []},
    {"title": "MCQs", "page_start": 23, "page_end": 25,
     "is_numbered": false, "expected_question_count": 25, "subsections": []}
  ]
}
```

**Nesting depth is UNLIMITED.** Excluded entries can have N levels of nested subsections if the PDF shows them. Preserve every printed level — do NOT flatten or collapse. At every level, parent's `expected_question_count` ≥ sum of children's counts.

### §8.5 Structural exam-paper parents (PYQ / question paper style)
When a heading's ENTIRE subtree contains ONLY question banks (subject or topic headings each carrying question lists, with zero theory paragraphs anywhere underneath), that heading is itself a named-bank parent — even if its label is generic ("Section A", "Section B", "Part I", "Paper 1", "Group A"). Place it in `excluded_sections` and nest the subject/topic banks under it as `subsections` per §8.4. **Children appear ONLY as nested subsections — do NOT also emit them as flat top-level excluded entries.**

**Worked example (MHT-CET PYQ paper, multi-column layout):**
PDF: top of paper has exam instructions (theory header). Below: Section A → Physics 50Q + Chemistry 50Q | Section B → Mathematics 50Q. Questions span multiple columns.

✅ CORRECT:
```
sections: [
  {"type": "chapter", "content_types": ["theory"],
   "title": "MHT-CET PYQ 2024",
   "subsections": [...]}        ← captures instruction text per §11.4
]
excluded_sections: [
  {"title": "Section A", "expected_question_count": 100, "subsections": [
    {"title": "Physics", "expected_question_count": 50, "subsections": []},
    {"title": "Chemistry", "expected_question_count": 50, "subsections": []}
  ]},
  {"title": "Section B", "expected_question_count": 50, "subsections": [
    {"title": "Mathematics", "expected_question_count": 50, "subsections": []}
  ]}
]
```

❌ WRONG (do NOT duplicate children as flat entries):
```
excluded_sections: [
  {"title": "Section A", ...},
  {"title": "Physics", ...},     ← FLAT DUPLICATE of Section A's child
  {"title": "Chemistry", ...},   ← FLAT DUPLICATE
  {"title": "Section B", ...},
  {"title": "Mathematics", ...}  ← FLAT DUPLICATE
]
```

## §9. PAGE RANGES

### §9.1 page_start / page_end
`page_start` = page where heading is printed. `page_end` = page where this section's content ENDS in column reading order (regardless of column). If a section runs from middle of left column, through bottom of left column, then top of right column ending mid-right-col on the SAME page — `page_end` = that same page.

### §9.2 Shared-boundary rule (column-aware)
Sections CAN share a boundary page when both have content on it: 8.1 ending p8 + 8.2 starting p8 is correct. Two adjacent siblings can both have `page_end=8` if the later one is single-page on the shared boundary. This is NOT a real overlap. REAL overlap (e.g. 8.1=[5,8] + 8.2=[6,8]) is FORBIDDEN.

In multi-column: if 8.1 ends in left column of page X and 8.2 starts in right column of page X, both share page X — shared boundary applies normally.

### §9.3 Single-page sections
`page_start == page_end` is allowed for short sections, including sections fitting in a single column on one page.

### §9.4 Child within parent
`parent.page_start ≤ child.page_start ≤ child.page_end ≤ parent.page_end`. No exceptions.

### §9.5 No coverage gaps
Within a chapter, the union of direct children's page ranges MUST cover the chapter's full range. Internal gaps mean a section is missing — re-scan.

## §10. HEADERLESS CONTENT FALLBACK

### §10.1 The rule
If content (theory OR numbered questions) exists but NO visible heading names it: use the chapter title (or document title if no chapter visible) as the wrapper section's `title`. Classify `content_types` correctly. All visible content goes under this wrapper. NEVER skip. NEVER fail.

### §10.2 Top-of-FIRST-column questions case (the multi-column failure mode)
A multi-column page shows 4–5 numbered questions at the top of the LEFT (first) column, then a heading mid-column or in a later column, then more questions. Without explicit handling, the top 4–5 questions are lost.

Fix: emit ONE Cat A wrapper using chapter/document title with `expected_question_count = number of LOOSE questions above the named heading in column reading order` (those questions only — see §11.2 for wrapper count semantics). The named heading becomes its own sibling Cat A section carrying its own questions' count. Apply same logic for 3-column layouts.

### §10.3 Anti-placeholder guard
NEVER emit empty chapter wrapper (`content_types=[]`, `expected_question_count=0`, `subsections=[]`, no real content) when ANY printed instructional content exists. If truly no structure identifiable, populate `extraction_notes` AND emit the §10.1 fallback wrapper.

## §11. PYQ / PURE QUESTION-BANK PAPERS (common in multi-column prep books)

### §11.1 Paper types
Two flavors:
- **Mixed textbook**: chapter has theory sections + worked examples + end-of-chapter exercises. Standard rules apply.
- **Pure question paper / PYQ**: dense Q-bank book or actual exam paper. May have theory header at top + question banks below. See §11.2–§11.4.

### §11.2 Wrapper Cat A vs Cat B decision (THE single test)
The chapter wrapper is EITHER Cat A or Cat B, NEVER both. Decide by ONE boolean:

**"Does this chapter contain ANY theory text ANYWHERE — loose on the wrapper, in subsections, in intro headers, in instructions, or in any Cat B section?"**

- **YES → Cat B `["theory"]`** (with optional `"figures"`). The most common case. Theory text exists somewhere → wrapper is Cat B regardless of how many question banks also exist. Examples: standard textbook with theory sections + end-of-chapter exercises; MHT CET PYQ with instruction header at top + question banks below.
- **NO → Cat A `["questions"]`** (with optional `"figures"`). The chapter is a pure question-bank container. Examples: ACFROG-style pure-Q book where the chapter contains only Practice Questions in `excluded_sections` and no theory paragraphs anywhere; Electricity multi-column with 4 loose Qs at top + Level 2 bank below (no theory anywhere in chapter).

When wrapper is Cat A `["questions"]`: `expected_question_count` = count of LOOSE questions directly on the wrapper (NOT including questions inside subsections or excluded banks). If zero loose Qs on the wrapper (all Qs live in excluded children, like ACFROG) → set `expected_question_count = 0`. The Cat A tag still applies because the chapter's overall content is questions, not theory.

### §11.3 Numbered sections inside vs outside named banks
Pure Q-bank prep books (dense 2-col or 3-col) may have numbered sections (4.1, 4.2, 4.3) appearing at chapter top level OR as sub-headings inside named banks. Placement is decided by §3 (parent attribution, accounting for column flow §3.0), NOT by numbering format.

CASE A — Numbered sections at chapter top-level (siblings of named banks):
```
sections: [chapter, subsections: [
  {Cat B "Chapter Outline"}, {Cat B "Quick Review"},
  {Cat A "4.2 Evaluation"}, {Cat A "4.3 Properties"}
]]
excluded_sections: [
  {Classical Thinking, subsections=[]},
  {Critical Thinking, subsections=[]}
]
```

CASE B — Numbered sections nested inside named banks (Indefinite Integrals):
```
sections: [chapter, subsections: [
  {Cat B "Chapter Outline"}, {Cat B "Quick Review"}
]]
excluded_sections: [
  {Classical Thinking, subsections: [
    {"4.1 Fundamental theorem"}, {"4.2 Evaluation"}, {"4.3 Properties"}
  ]},
  {Critical Thinking, subsections: [...]}
]
```

**CRITICAL — preserve PDF visual nesting.** If "4.2 Evaluation" appears UNDER "Classical Thinking" in column reading order, then 4.1, 4.2, 4.3 stay NESTED inside Classical Thinking in excluded_sections. NEVER pull a numbered sub-heading out of its parent bank to inline.

For each excluded bank (and its sub-banks), count questions across ALL columns per §7.7 — one bank, not one per column.

The difference between Case A and Case B is decided by §3 (which heading precedes the numbered section in column reading order) — NOT by numbering format itself.

### §11.4 PYQ paper with theory header at top (THE MHT-CET / NEET / JEE pattern)
A question paper with instructions / syllabus / explanatory text at the top of the paper, followed by question banks (often grouped under structural labels like Section A / Section B per §8.5).

Per §0.5 #5: theory at top SURVIVES — never drop it.
- Chapter wrapper: Cat B `["theory"]` carrying the header content.
- Question banks below: go to `excluded_sections` (no theory after them = excluded per §0.5 #4).
- Structural section labels (Section A/B/etc.): emit as excluded parent banks with subject/topic banks nested per §8.5.

See §8.5 worked example for the canonical MHT-CET shape.

## §12. SEQUENCE PRESERVATION + SIBLING ORDERING

### §12.1 Reading order (overall)
Preserve PDF reading order. In multi-column: siblings appear in COLUMN reading order (§3.0) — left col → middle col (if 3) → right col, top-to-bottom each, with spanning headings interleaved at their visual position. End-of-chapter content (`excluded_sections`) MUST appear AFTER all inline content (`sections`), never mixed.

### §12.2 Sibling ordering rule (CRITICAL — fixes the "Example 3.8 before Point B" failure mode)
Within ANY parent's `subsections` array, order children by:
1. `page_start` ascending (lower page first), THEN
2. for siblings on the SAME page: by physical position in column reading order (left col → middle col → right col, top-to-bottom each).

Never reorder by section number, label type, or perceived importance. Examples don't automatically come first or last; their array index reflects exactly where they print in column reading order.

**Worked example (Percentage book, single-column or multi-column):**
PDF page 7, column reading order: "Point A" theory → "Point B" theory → "Example 3.8".

CORRECT: `parent.subsections = [Point A, Point B, Example 3.8]`
WRONG: `parent.subsections = [Example 3.8, Point A, Point B]` (Examples grouped first by label type)

When a child's title sits near a page or column boundary, use the title's `page_start` per §3.4.

## §13. ID FORMAT

### §13.1 Numbered IDs
Use section number as id, no "Chapter" prefix: Chapter 8 → `"8"`; Section 8.4.1 → `"8.4.1"`; Chapter 8A → `"8a"`.

### §13.2 Unnumbered IDs (stable slugs)
`"intro"`, `"summary"`, `"glossary"`, `"index"`, `"bibliography"`. Biographical: `"bio-ramachandran"`, `"about-newton"`. Cat A wrappers: `"<chapter>-solved-examples"`, `"<chapter>-practice-questions"`.

### §13.3 Uniqueness
IDs MUST be unique across entire document. If a slug could collide, prefix with parent id: `"8.4-didyouknow"`.

### §13.4 Duplicate-named sections
If the SAME heading appears multiple times in the PDF (e.g. "Quick Recap" at both chapter start AND chapter end; "Activity 1" appearing 3 times across a lab manual; "Try It" repeating in every section), each instance is its OWN section with distinct page ranges — NEVER merged into one. Use contextual suffixes to keep ids unique: `quick-recap-intro` and `quick-recap-summary`; `activity-1-experiment` / `activity-1-observation` / `activity-1-conclusion`; `try-it-3.1` / `try-it-3.2`. Preserve their print order in the parent's `subsections` array (§12). Per-instance ids must remain unique across the entire document.

## §15. PRE-FLIGHT SELF-CHECK (run before emitting; fix every failure)

1. Every section has all required fields (§2.3), all formats correct (§2.4).
2. No mixed `["theory","questions"]`. No forbidden tokens. No `content_types=[]` or missing. No `uuid` emitted.
3. Every X.Y.Z nested under X.Y. No level skips.
4. Every section title points to verbatim text on a specific page.
5. `page_end ≥ page_start` everywhere; both integers.
6. Child page range fits within parent (§9.4). No coverage gaps within chapter (§9.5).
7. Siblings may share boundary page (§9.2); real overlap forbidden.
8. **POSITION TRUMPS LABEL** verified (§0.5 #4, §8.3): every item in `excluded_sections` has ZERO theory printed after it in column reading order. Every bank-named section followed by theory is INLINE Cat A under its theory parent, not excluded.
9. Numbered sections (X.Y) placed by §3 parent rule — INLINE if their preceding heading is the chapter; NESTED inside the bank's excluded entry if their preceding heading is a named bank (§11.3 Case B). NEVER pull a numbered sub-heading out of its parent bank to inline.
10. Index / Bibliography / Glossary / Appendix / Summary in `sections` as Cat B theory, NOT excluded.
11. **Sibling ordering** (§12.2): within every parent's `subsections` array, children are in PDF column reading order (sort by page_start, then position in column reading order on same page). No reordering by label type.
12. **Structural labels nested correctly** (§8.5): when "Section A" / "Part I" / "Paper 1" wraps subject/topic banks, the children appear ONLY as nested subsections — NOT duplicated as flat top-level excluded entries.
13. **Theory survived** (§0.5 #5, §11.4): if the PDF contains any theory content (instructions, intro, explanatory text), it appears in `sections` — typically on the chapter wrapper as `["theory"]`. Empty `sections: []` only when the PDF truly has zero theory anywhere.
14. Question count consistency: at every level, parent's `expected_question_count` ≥ sum of subsections' counts. Parent's count CAN be higher (when parent has questions printed directly under it but not under any subsection). Parent's count must NEVER be less than the sum. Wrapper `expected_question_count` (§11.2) = LOOSE questions on the wrapper only, never includes nested or excluded counts.
15. JSON valid: no trailing commas, balanced brackets, no markdown fences.
16. (Multi-column) Reading order applied: left col → (middle col) → right col, top-to-bottom each. Spanning headings respected as parents.
17. (Multi-column) Question banks crossing columns counted as ONE bank with combined `expected_question_count` (§7.7).

Return only the JSON.
