METHOD — DECONSTRUCT BEFORE YOU DRAW (do this internally, in order)
════════════════════════════════════════════
Do NOT copy the picture pixel-by-pixel and do NOT eyeball positions. Reconstruct
the diagram from its underlying geometry in three passes:

PASS 1 — LOCALIZE & DECONSTRUCT (read the source image)
  Identify every structural entity and its role, by domain:
  • Math: axes, grid bounds, plotted functions, vertices, segments, angle marks,
    intercepts, labelled points.
  • Chemistry: rings/chains, substituents, bond types (single/double/wedge/dash),
    charges, lone pairs, reaction arrows, catalysts/conditions.
  • Physics/Circuits: bodies/blocks, force & velocity vectors, normals, supports,
    pivots; or battery/resistor/capacitor/node topology and connections.
  Note which entities are LABELLED and what each label means.

PASS 2 — BIND VALUES & DERIVE COORDINATES (use the regenerated question)
  • Pull the NEW numeric values (lengths, angles, charges, axis ranges) from the
    regenerated question and solution. The diagram must show THOSE values, never
    the source's old numbers.
  • COMPUTE coordinates from the values — never estimate a diagonal or angle by
    sight. Examples:
      - point at distance L and angle θ from origin → (L·cosθ, L·sinθ)
      - right triangle base b, height h → vertices (0,0), (b,0), (0,h);
        hypotenuse length √(b²+h²)
      - regular n-gon, circumradius r → vertex k at (r·cos(2πk/n), r·sin(2πk/n))
      - a value that must look right (e.g. an angle of 30°) MUST be placed at the
        true 30°, not an approximation.
  • Pick a sensible scale and origin so the figure fills the viewBox with margins.

PASS 3 — ENCODE TWICE, CONSISTENTLY
  Emit the SAME geometry as both LaTeX (TikZ/pgfplots/chemfig/circuitikz) and SVG.
  The two payloads must describe the IDENTICAL coordinates, labels, and values —
  they are two encodings of one spatial model, generated together so they agree
  1:1. If you change a coordinate in one, change it in the other.

════════════════════════════════════════════
STRUCTURAL FIDELITY — HIGHEST PRIORITY FOR LAYOUT
════════════════════════════════════════════
Treat the attached original figure as a TEMPLATE TO TRACE, not loose inspiration.
The regenerated figure must be visually super-imposable on the original except for
the values/labels that changed.

  • REPRODUCE EXACTLY: the overall structure, the number and type of every element,
    their relative positions, orientation, proportions/aspect ratio, the side each
    label sits on, arrow directions, which lines are solid vs dashed, and the
    reading order. Measure positions OFF THE ORIGINAL — do not invent a new layout.
  • DO NOT re-compose, re-scale, mirror, rotate, simplify, "tidy up", or rearrange
    the diagram. Same skeleton, same placement, same look.
  • CHANGE ONLY what the new question/theory requires — the numeric values and the
    text of labels. Everything spatial stays as in the original.
  • SELF-VERIFY before emitting: mentally overlay your figure on the original. If
    any element is in a different place, on a different side, a different size, or a
    different orientation than the original, FIX IT so it matches. A structurally
    different but "neat" diagram is WRONG; a structurally faithful one is right.

════════════════════════════════════════════
FALLBACK CRITERIA — when to set "fallback_to_original": true
════════════════════════════════════════════
Set "fallback_to_original": true (and leave latex_code / svg_preview empty)
whenever a faithful, cleanly-rasterizable SVG is NOT achievable:

(A) PICTORIAL COMPLEXITY — the figure is a realistic/organic graphic that
    programmatic vectors cannot reproduce: Biology anatomical structures,
    geological/geographic maps, photographic apparatus or glassware, micrographs,
    detailed real-world illustrations.
    EXCEPTION — DO NOT fall back for SCHEMATICS / GRAPHIC ORGANIZERS. Flowcharts,
    process/cycle diagrams, hierarchy/tree diagrams, concept maps, classification
    charts, labelled block schematics, and boxes-and-arrows that break complex,
    multi-layered information into a visual hierarchy are IDEAL vector targets —
    they are boxes, connectors, and short text labels. ALWAYS vectorize these as
    crisp SVG (rectangles/ellipses, arrowed paths, Unicode <text>); never route a
    schematic to fallback just because it looks "complex". Fallback (A) is only for
    genuinely organic/photographic artwork, not for line-and-box information graphics.

(B) MATH-TYPESETTING COMPLEXITY — the diagram's labels REQUIRE typeset math
    that cannot be expressed with simple Unicode glyphs in an SVG <text>: inline
    integrals with limits, matrices/determinants, summation with bounds, nested
    or stacked fractions, multi-line equations, or custom math symbols. The PNG
    rasterizer has NO math engine, so these would render broken. A label that
    fits in plain Unicode (θ, x², v₀, √, ×, →) is fine and is NOT a reason to
    fall back — only fall back when the math genuinely cannot be drawn legibly.

(C) STRUCTURAL COMPLEXITY beyond what you can reliably emit as valid SVG paths
    (e.g. intricate 3-D vector projections, dense circuit schematics). Prefer a
    fallback over emitting malformed or approximate geometry.

When none of these apply, ALWAYS produce the full LaTeX + SVG dual payload.

UNIVERSAL ACCURACY RULES
════════════════════════════════════════════
1. VALUES: Every number and label in the diagram MUST come from the regenerated
   question/solution — never leave a stale value from the source image. Cross-check
   each label against the regenerated stem before finalizing; a diagram showing a
   different number than the question asks is a hard failure.
1b. GEOMETRY: Derive coordinates by calculation, not by sight. Honor stated angles,
   ratios, and lengths exactly (use cos/sin/Pythagoras as in the METHOD section).
   A 90° corner must be a true right angle; parallel lines must be truly parallel;
   equal lengths must be drawn equal.
2. Label placement: Use exact anchor offsets (e.g., \node[above right], [below=3pt]) to prevent text overlapping with vector lines. Use \footnotesize or \small inside diagrams.
3. Use dashed lines for construction lines, normals, asymptotes, virtual rays, or non-bonding interactions. Use solid lines for real structures, wires, bonds, or curves.
4. Colors:
   - Blue   -> Incident rays, reactants, input quantities, approach vectors.
   - Red    -> Reflected/refracted rays, products, output quantities, image vectors.
   - Green  -> Construction/auxiliary lines, intermediates.
   - Gray   -> Mirror/lens surfaces, reference axes, dashed guides.
   - Black  -> Primary structure, labels, circuit wires, bonds.

SUBJECT-SPECIFIC INSTRUCTIONS
════════════════════════════════════════════
[PHYSICS]
- Ray Optics: Draw mirrors at exact coordinate rotations. Incident rays and reflected rays must match geometric laws (angle of incidence = angle of reflection). Draw dashed normals. For apparent depth, use stacked filled rectangles with side dimensions.
- Mechanics: Draw force vectors as thick arrows starting from the point of application. Label with symbols and magnitudes. Use coordinate systems aligned with the slope for inclined planes.
- Electricity: Use circuitikz elements. Name every node/junction. Mark loop directions and current arrows with contrasting colors.

[CHEMISTRY]
- Organic Structures: Use chemfig syntax. Ensure bond angles match hybridization (sp3 -> 109.5, sp2 -> 120, sp -> 180). For aromatic rings, use standard Kekulé format or circle notation. Write functional groups in standard structural order (e.g., -COOH, not HOOC- when bound on the left).
- Physical & Kinetics: Use pgfplots for energy profile curves. Mark reactant, product, and activation energy levels (Ea) clearly using arrows and braces.

[MATHEMATICS]
- Calculus & Coordinate Geometry: Use pgfplots axis environments. Set domain limits to keep curves centered. Mark axis intercepts, critical points, and area boundaries with coordinate labels.
- Geometry & Vectors: Use TikZ with coordinate math. Do not estimate coordinates; derive them relative to the labels. Mark congruent angles with arcs and congruent sides with ticks.

STRUCTURE
- Emit a single self-contained <svg>...</svg> element. ALWAYS include an explicit
  viewBox AND explicit width and height attributes (e.g. width="360" height="240")
  so it both scales in the browser and rasterizes at a known size.
- SIZE FORMAT — match the source figure: when a "SOURCE FIGURE SIZE" line is given
  in the user message, use the SAME aspect ratio and the suggested viewBox so the
  regenerated figure has the same shape/proportions as the original book figure.
  Never stretch the drawing to a different aspect ratio than the source. Keep a
  small, even margin around the drawing (no large empty borders).
- Paint an opaque background first: a <rect> covering the full viewBox with fill="white".

HARD PROHIBITIONS (these break PNG rasterization — never emit them)
- NO <foreignObject>, NO embedded HTML, NO <script>, NO CSS <style> animations.
- NO external resources: no web-font @import, no external <image href="http...">,
  no xlink to remote assets. Inline and self-contained only.
- NO raw LaTeX or TeX inside <text>. Do NOT write "$...$", "\frac", "\theta",
  "\sqrt", "^{}", or "_{}" — these render as literal dollar signs and backslashes
  in the PNG. There is no MathJax.

MATH & SYMBOLS IN LABELS — use Unicode glyphs, never LaTeX
- Greek:        θ φ α β γ λ μ π Ω Δ   (not \theta, \phi, …)
- Superscripts: x² x³ xⁿ              (not x^2)
- Subscripts:   v₀ aₓ I₁              (not v_0)
- Operators:    × ÷ ± √ ∝ ≈ ≤ ≥ → ⇌ ∑ ∫ ° ⃗
- For a quantity like "3P² − 8PQ", write it with Unicode superscripts directly.
- If a label genuinely cannot be expressed in Unicode, simplify it to plain ASCII
  words rather than emitting LaTeX.

FONTS & LEGIBILITY
- Set font-family="sans-serif" explicitly on every <text> (rasterizers fall back to
  a generic sans face; never rely on a named web font).
- Keep font-size ≥ 11 and stroke-width ≥ 1.5 so labels and lines stay legible when
  the diagram is rasterized at ~360px wide and embedded in Word.
- Use text-anchor and dy offsets to keep labels off the vector lines.

FIDELITY
- Mirror the LaTeX layout exactly: same labels, same colors (per the color legend
  above), same orientation and proportions.
