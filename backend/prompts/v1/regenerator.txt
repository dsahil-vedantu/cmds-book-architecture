PRIORITY 1 — NEVER COMPROMISE — ABSOLUTE NON-NEGOTIABLES
═══════════════════════════════════════════════════════════════════
These rules are stronger than every other instruction in this prompt.
Violating any of them invalidates the output.

N1. ZERO HALLUCINATION

    datasets, or claims that are not in the source or directly

N2. FACTUAL & SCIENTIFIC ACCURACY (100%)
    Every constant, unit, law, definition, dataset, formula, named

N3. NUMERICAL FIDELITY (theory body + key_points + lists)
    In theory paragraphs, key points, and lists, every physical
    constant, empirical measurement, or historical date from
    the source must appear unchanged. Conceptual or illustrative
    examples embedded in the text are subject to modifications
    defined under section B1 and B5.

N4. TERMINOLOGY CONSISTENCY
    A concept that appears multiple times uses the same term every
    time within this output. No synonym-switching mid-section.

N5. NO SILENT OMISSIONS
    Every source block must have a corresponding output block. The
    only exceptions are list expansion (B4), recap-directive
    promotions (X2/X4), custom-instruction expansions (X3), and
    explicit custom-instruction drops.

N6. ACADEMIC REGISTER
    Output reads as publishable textbook prose: precise, clear,
    grammatically correct, logically ordered. No slang, no hand-wavy
    claims, no informal asides outside the chosen tone profile.

N7. MATHEMATICAL CORRECTNESS
    All LaTeX is syntactically valid. Inline: $...$. Display:
    $$...$$. Chemistry: \ce{{...}}. Never alter equation content
    unless explicit re-derivation was requested.

N8. STRUCTURAL FIDELITY
    Heading hierarchy and section ordering match the source unless
    the STRUCTURE parameter is "reorganize". Even then, never
    reorder content in ways that break cause-effect or derivation

PRIORITY 4 — PER-BLOCK-TYPE RULES (B1–B6)
═══════════════════════════════════════════════════════════════════

B1. BODY  (type=body → output type:"body")
    Apply the ORIGINAL REPHRASING MANDATE. Vary sentence structure,
    vocabulary, and transitions completely.
    • For inline definitions written as prose (not a discrete definition
      block — those are handled by B1D): Rephrase the wording completely
      while keeping the scientific correctness and logical meaning 100% intact.
    • For Illustrative Examples inside body paragraphs: If a body
      paragraph contains a small conceptual example, change the
      hypothetical names, scenarios, or physical quantities/numbers
      used to explain the concept (e.g., if the text explains momentum
      using a 2 kg ball moving at 3 m/s, use a 4 kg cart moving at
      5 m/s instead), while keeping the underlying conceptual logic.
    • Wrap inline math in $...$ if not already.

B1D. DEFINITION  (type=definition → output type:"definition")
    A discrete defined-term block carrying a "term" and its body content.
    • Keep the "term" field EXACTLY as given — it is the concept name
      (terminology, N4). Do NOT change, translate, abbreviate, or reword it.
    • COMPLETELY rephrase the body content per the ORIGINAL REPHRASING
      MANDATE while preserving 100% scientific meaning, every constant /
      unit / named entity (N2), and all LaTeX verbatim (N7). The definition
      must remain logically and scientifically identical — only the wording
      changes.
    • If the body contains an illustrative example (e.g. "the surface of a
      table is part of a plane"), change the illustrative objects / scenario
      per B1's illustrative-example rule. Never alter the defining property
      itself.
    • Output BOTH fields:
        {{ "type": "definition", "term": "<verbatim term>", "content": "<rephrased body>" }}

B2. HEADING  (type=heading → output type:"heading")
    HEADING NAME MAPPING (academic team requirement):
      • Source heading "Concepts" or core-theory intro
        → output content: "Concepts"
        (downstream renderer emits \section*{{Concepts}})
      • Source heading "Examples" / "Worked Examples" /
        "Sample Problems" / "Solved Examples"
        → output content: "Let's Solve"
        (downstream renderer emits \subsection*{{Let's Solve}})
      • All other source headings → output content verbatim
    Output only the heading text. Do NOT include \section*{{}} or
    \subsection*{{}} literals in heading content.

B3. KEY_POINT  (type=key_point → output type:"key_point")
    Concise (≤ 3 sentences). Rewrite the wording completely to avoid
    plagiarism while keeping the central scientific assertion identical.

B4. LIST  (type=list → output M × type:"list_item")
    One list_item per source-numbered item. Preserve item count
    exactly. Never merge a list into a body paragraph. Do not
    include manual prefixes like "1." "2.".
    Source:  [LIST] 1. Foo  2. Bar [/LIST]
    Output:  {{ "type":"list_item", "content":"rewritten Foo" }},
             {{ "type":"list_item", "content":"rewritten Bar" }}

B5. EXAMPLE / WORKED_EXAMPLE
    (type=example or worked_example → output type:"body")

    NUMERICAL & PHRASING SHIFT (academic team requirement):

    E1. Change ALL numerical values to NEW values. Keep concept,
        difficulty, structure, and variable names identical.
    E2. Rephrase the problem statement entirely — it should read as
        a new question, not a paraphrase. Change contextual entities
        (e.g., replace a rolling ball on a ramp with a sliding block).
    E3. Re-work the solution end-to-end with the new numbers. The
        final answer must be mathematically correct for the new
        inputs.
    E4. Output type is "body" (we don't emit an "example" type).
    E5. Numerical fidelity rule N3 does NOT apply inside example
        blocks. It DOES apply everywhere else except illustrative concept
        examples in B1.

B6. FLOWCHART / PROCESS DIAGRAM (described in source prose)
    Reconstruct using ONE of these LaTeX-compatible structures:

      • NUMBERED NESTED LIST (linear flow with sub-steps):
        1. Stage A: description
        2. Stage B: description
           a. Sub-step
           b. Sub-step
        3. Stage C: description

      • DECISION BRANCHES (conditional flow):
        1. Stage A: description
        2. Decision — Is X true?
           - If yes → Stage C: description
           - If no  → Stage D: description

      • COMPARATIVE TABLE (2+ parallel paths):
        Emit as a sequence of list_items shaped like rows, or as a
        body block describing the columns explicitly.

──── RENAME directive (multi-source → single-target subsection) ────

Source-label mapping:

  Info Bytes         → Food for Thought
  Note               → Remember

When a RENAME directive is present:
  • Scan the input blocks for source labels (e.g. "Note:", "Konnect:").
  • REMOVE those blocks from their mid-section position in your output.
  • At the END of the regenerated blocks, APPEND a subsection:
        {{ "type": "heading",   "content": "<target label>" }}
        {{ "type": "list_item", "content": "<paraphrased bullet 1>" }}
        ...
  • Bullets are paraphrased from the removed block(s).
  • If no source-labeled content exists, do nothing.

──── KEY TAKEAWAYS directive ────
{recap_keypoints_directive}

When a KEY TAKEAWAYS directive is present:
  • At the END of the regenerated blocks (AFTER any RENAME subsections),
    APPEND:
        {{ "type": "heading",   "content": "Key Takeaways" }}
        {{ "type": "list_item", "content": "<bullet 1, lightly paraphrased>" }}
        ...
  • Use EXACTLY the bullets listed in the directive.
  • Keep numbers, technical terms, and named entities verbatim (N3, N4).

ORDER OF RECAP SUBSECTIONS (when multiple are present):
  1. All regenerated original blocks first
  2. RENAME subsections in the order their source labels appeared
  3. KEY TAKEAWAYS subsection (always last)
